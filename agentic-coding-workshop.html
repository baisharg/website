🚨 El Problema

Los LLMs tradicionales tienen dos problemas fundamentales para desarrollo de software:

📉 Degradación con Contexto Largo

El benchmark LoCoDiff (enero 2025) muestra que incluso el mejor modelo disponible, Sonnet 4.5, degrada significativamente con contextos largos:

El resultado: De 96% de precisión con 2-21K tokens a 64% con más de 60K tokens. Cuando le das todo tu codebase al modelo, se ahoga en información.

📄 Source: LoCoDiff Benchmark

🧠 Sin Memoria a Largo Plazo

Un estudio de METR (julio 2025) demostró que developers expertos con 2+ años de experiencia en sus propios proyectos fueron 20% MÁS LENTOS con AI que sin ella.

¿Por qué? Porque el LLM arranca de cero cada vez, sin el contexto que los developers tienen en su cabeza. Los expertos predijeron que serían 39% más rápidos, pero la realidad fue lo opuesto.

📄 Source: METR AI R&D Study (July 2025)

✅ La Solución: BMAD

BMAD es un método de desarrollo de software con AI donde un equipo de 8 agentes especializados trabaja como un equipo real: planean el proyecto completo antes de codear, después ejecutan story por story con validaciones continuas.

Cada agente es el mismo LLM con un prompt especializado y contexto quirúrgicamente preciso. No hay memoria mágica - hay documentación estructurada que los agentes leen exactamente cuando la necesitan.

BMAD resuelve los problemas de degradación y falta de memoria con tres principios fundamentales:

🔪 Sharding

Partir documentos grandes en pedazos pequeños y enfocados. En vez de darle al Developer un PRD de 10,000 tokens, le das un "shard" de 500 tokens con exactamente lo que necesita para esta story.

Resultado: Cada agente opera en la zona de 96% de precisión (< 5K tokens).

👥 Agentes Especializados

8 agentes con roles específicos: Product Manager, Architect, Developer, QA, etc. Cada uno tiene instrucciones especializadas y lee SOLO lo que necesita para su tarea.

Resultado: Contexto quirúrgicamente preciso para cada decisión.

🎯 Documentación Estructurada

Planning riguroso antes de codear: PRD completo, arquitectura definida, stories secuenciales. Esto crea la "memoria" que los LLMs no tienen.

Resultado: Los agentes tienen contexto profesional, no tienen que adivinarlo.

💡 ¿Por qué BMAD funciona?

1. Contexto Optimizado: Los expertos en el estudio METR usaron Cursor con Claude Sonnet 3.5 en codebases grandes y fueron 20% más lentos. BMAD hace lo opuesto: cada agente recibe SOLO lo que necesita. PM lee requirements docs (2K tokens), Architect lee PRD + preferencias técnicas (3K tokens), Developer lee UN epic específico (1.5K tokens). Esto mantiene la precisión alta evitando la degradación por contexto excesivo.

2. Validaciones y Transparencia: BMAD tiene múltiples checks de calidad integrados. El PO valida que los story drafts sean coherentes con el PRD. El QA testea el trabajo del Developer exhaustivamente. La documentación es completamente legible por humanos, y el humano aprueba cada story antes de que se implemente.

3. Elicitación Avanzada: Los agentes usan técnicas de elicitación para hacer las preguntas correctas al usuario, empoderándolo para construir exactamente lo que quiere construir. No es solo generar código - es un proceso colaborativo de descubrimiento y validación continua.

⌨️ Cómo Funcionan los Comandos

📝 Sintaxis de Comandos

Los agentes se invocan a través de comandos en tu herramienta de AI. La sintaxis depende de la herramienta que uses:

En Claude Code / OpenCode:

/pm *create-prd here's some rough notes @notes.md /architect *create-full-stack-architecture @prd.md /dev *develop-story @docs/stories/1.1.md

En Cursor / Windsurf:

@PM Create prd, here's some notes @notes.md @architect Design the system architecture, here's the @prd.md @dev *develop @docs/stories/1.1.md

🔄 Cómo Cambiar de Agente

Para invocar un agente diferente:

  1. Limpia el contexto con: /clear (Claude Code) o /new (OpenCode)
  2. Invoca el agente con su comando de inicialización (ej: /pm)
  3. El agente toma control con su prompt especializado y contexto preciso

💡 Importante: Cada agente es el mismo LLM (Claude, GPT, etc.) pero con instrucciones especializadas. Limpiar el contexto asegura que el nuevo agente arranque limpio sin confusión de conversaciones anteriores.

📐 Fase 1: Planning

Diseño completo antes de codear. Se hace UNA VEZ al principio del proyecto.

💡 Todo el planning se ejecuta en la terminal con los comandos de cada agente.

Workflow Core (4 pasos obligatorios)

1
📋 PM: PRD Creation
El Product Manager crea el Product Requirements Document siguiendo un proceso paso a paso:
  • Proceso interactivo: El PM hace preguntas al usuario sección por sección del PRD
  • Features y epics (con stories MUY cortas - solo títulos)
  • Requirements funcionales y no funcionales
  • Priorización MVP vs roadmap
  • El usuario revisa y aprueba cada sección antes de continuar
💡 Nota importante: Las stories en el PRD son muy breves (solo títulos). El Scrum Master las expandirá más tarde con todos los detalles.
/pm *create-prd
2
🏗️ Architect: System Design
El Architect lee el PRD y diseña la arquitectura técnica siguiendo un proceso paso a paso:
  • Proceso interactivo: El Architect hace preguntas al usuario sección por sección
  • Tech stack (frontend, backend, base de datos)
  • Estructura de código y carpetas
  • APIs y contratos entre componentes
  • Consideraciones de escalabilidad, seguridad, performance
  • El usuario revisa y aprueba cada sección antes de continuar
/architect *create-full-stack-architecture
3
✅ PO: Master Checklist
El Product Owner valida que PRD y Arquitectura estén alineados:
  • Coherencia entre requirements y diseño técnico
  • Viabilidad de la arquitectura para los features
  • Identificación de gaps o contradicciones
/po *execute-checklist-po
⚠️ Si no alinea: Se refinan documentos hasta que todo tenga sentido. No se avanza hasta que haya validación completa.
4
🔪 PO: Sharding
El Product Owner shardea los archivos docs/prd.md y docs/architecture.md en epics y stories manejables:
  • El PO ejecuta un programa de terminal que divide automáticamente los documentos
  • Divide documentos grandes en shards pequeños (< 2K tokens cada uno)
  • Cada shard es un epic o story con contexto enfocado
  • Todos los archivos son Markdown guardados en docs/ en la raíz del proyecto
  • Ejemplo: PRD de 10K tokens → docs/epics/epic-1-auth.md, docs/epics/epic-2-dashboard.md, etc.
/po *shard-doc docs/prd.md
/po *shard-doc docs/architecture.md
✅ Resultado: Backlog de stories shardeadas en docs/epics/ y docs/stories/, listas para desarrollo iterativo.

📌 Agentes Adicionales (según necesidad)

• Business Analyst: Para proyectos greenfield donde necesitás investigación de mercado y brief inicial.
• UX Expert: Para apps con UI - crea specs de diseño y prompts para herramientas como V0/Lovable.
• QA: Para risk profiling de áreas de alto riesgo antes de empezar desarrollo.

🎯 Al Final de Planning tenés:

✅ PRD completo y validado
✅ Arquitectura definida y alineada con PRD
✅ Backlog de stories shardeadas y priorizadas
✅ Contexto estructurado para todo el equipo

No tocaste una línea de código todavía. Y eso está bien - el planning correcto te ahorra semanas de refactoring después.

🔄 Fase 2: Development Loop

Implementación iterativa story por story. Este loop se repite hasta completar el proyecto.

Workflow Core (7 pasos iterativos)

1
📝 SM: Review Previous Notes
El Scrum Master revisa las notas de la story anterior (si existen):
  • Qué funcionó, qué no funcionó
  • Decisiones técnicas importantes
  • Feedback del Developer y QA
  • Aprendizajes para aplicar en la siguiente story
/sm *review-notes @docs/stories/1.0.md
💡 Loop de Aprendizaje: Las notas de hoy informan el draft de mañana. El equipo mejora continuamente.
2
📋 SM: Draft Next Story
El Scrum Master prepara la siguiente story desde el epic shardeado:
  • Lee SOLO el epic relevante (no todo el PRD)
  • Lee las partes relevantes de la arquitectura
  • Lee las partes relevantes del PRD original
  • Crea tareas secuenciales claras: "Primero esto, después esto"
  • Define criterios de aceptación específicos
  • Incluye contexto necesario sin ahogar en información
/sm *draft story 1.1
3
✅ PO: Validate Story Draft
El Product Owner valida el story draft contra el PRD:
  • Verifica que el draft sea coherente con el PRD original
  • Asegura que todos los requirements estén cubiertos
  • Identifica gaps o contradicciones
  • Da consejos de mejora al Scrum Master si es necesario
/po *validate story @docs/stories/1.1.md
✅ Check de Alineación: Esto asegura que el Developer implemente exactamente lo que se definió en planning.
4
💻 Dev: Implementation
El Developer implementa la story completa:
  • Código de producción: Sigue la arquitectura y el spec de UX
  • Tests unitarios: Para cada función/componente
  • Tests de integración: Para flujos entre componentes
  • Tests E2E: Para user flows completos (con Playwright MCP si es app web)
  • Validaciones: Manejo de errores, edge cases
/dev *develop-story @docs/stories/1.1.md
5
🔍 QA: Test Story Thoroughly
El QA testea exhaustivamente el trabajo del Developer:
  • Ejecuta todos los tests: Unitarios, integración, E2E
  • Testing manual: User flows, edge cases, escenarios de error
  • Verifica requirements: La implementación cumple los criterios de aceptación
  • Risk profiling: Identifica áreas de riesgo (seguridad, performance)
  • Emite veredicto: PASS ✅ / CONCERNS ⚠️ / FAIL ❌ / WAIVED 🔓
/qa *review @docs/stories/1.1.md
📋 Quality Gates:
• PASS ✅: Todo funciona, sin problemas críticos
• CONCERNS ⚠️: Problemas no críticos que el equipo debe revisar
• FAIL ❌: Problemas críticos que deben arreglarse (seguridad, tests P0 faltantes)
• WAIVED 🔓: Problemas aceptados explícitamente por el equipo
✅ Check Crítico: El QA asegura que el código no solo funciona, sino que cumple estándares de calidad profesional.
6
🔧 Dev: Fix According to QA Review
El Developer corrige el código según el feedback del QA:
  • Revisa los resultados del QA (PASS/CONCERNS/FAIL)
  • Implementa los fixes necesarios
  • Aborda todos los CONCERNS y FAIL items
  • Verifica que los cambios no rompan tests existentes
/dev Fix the issues from QA review @docs/stories/1.1.md
⚠️ Iterativo: Si el QA emitió FAIL, el Developer debe arreglar y el QA revisa nuevamente hasta obtener PASS o WAIVED.
7
🔄 Mark Done & Next Story
Marcar la story como completa y volver al paso 1 para la siguiente:
  • Actualizar el backlog (story marcada como done)
  • Revisar qué sigue en el backlog priorizado
  • Loop back al paso 1: SM revisa las notas que acabás de dejar
🔁 Iterativo: Story por story, commit por commit, hasta completar el proyecto.

🔁 El loop se repite: Volvés al paso 1 con la siguiente story

📌 Notas sobre el Proceso

• Validaciones Integradas: El PO valida el draft antes de development (paso 3), el QA testea después de development (paso 5 - incluye verificación de tests y commit), y el Developer arregla según feedback (paso 6). Estos checks mantienen el sistema alineado con los requirements y estándares de calidad.
• Flexibilidad de Calidad: El QA emite PASS ✅ / CONCERNS ⚠️ / FAIL ❌ / WAIVED 🔓. Vos decidís tu barra de calidad - si es un MVP rápido podés aceptar CONCERNS, si es software crítico no.
• Loop de Aprendizaje: El Developer documenta notas automáticamente durante el desarrollo (paso 4), y el SM las revisa en la siguiente story (paso 1). El equipo mejora continuamente.

💪 Ventajas del Loop Iterativo

✅ Siempre tenés una versión funcional: Después de cada story, el código funciona y está testeado.
✅ Flexibilidad para cambiar rumbo: Si después de 5 stories querés pivotar, podés. No estás comprometido a un plan gigante.
✅ Loop de aprendizaje: El equipo mejora story tras story basándose en notas de lo que pasó.
✅ Human in the Loop: Vos aprobás transiciones clave. Sos el tech lead que guía el equipo.

Instalación de BMAD

BMAD se instala por proyecto en el directorio del proyecto:

En la raíz de tu proyecto:
npx bmad-method install

Esto crea la carpeta .bmad-core/ con todos los agentes, templates, tasks, y checklists.

  • ✓ Instalación por proyecto (no global)
  • ✓ Todo versionable con git
  • ✓ Personalizable para tu proyecto
